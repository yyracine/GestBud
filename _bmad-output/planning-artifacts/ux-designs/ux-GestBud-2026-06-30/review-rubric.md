# Spine Pair Review — GestBud

## Overall verdict
The spine pair is strong on its core flows (UJ-1/2/3 are fully realized with named protagonist, numbered steps, climax, and failure paths) and on token/color discipline (every token resolves, contrast rules are stated, Do's/Don'ts are concrete). The critical gap is **Gestion Catégories / Paramètres**: an entire feature area with four dedicated FRs (FR-13–FR-16) has zero Component Patterns, zero State Patterns, and zero Key Flow coverage in EXPERIENCE.md, and zero visual spec in DESIGN.md — a downstream engineer has nothing to build against for category CRUD or the settings surface. A second tier of gaps sits in component coverage: five EXPERIENCE.md component-pattern rows (Sélecteur Catégorie, Sélecteur Date, En-tête sticky Reçu, Entrée Reçu groupé, Sélecteur de Période) have no matching visual spec in DESIGN.md, and conversely `nav-bar` and `input` exist as DESIGN.md tokens/components with no behavioral row in EXPERIENCE.md. These are fixable in a single pass without re-architecting either document.

## 1. Flow coverage — strong
Checked: PRD §2.3 defines UJ-1, UJ-2, UJ-3. EXPERIENCE.md Key Flows has Flow 1/2/3 mapped 1:1 to UJ-1/2/3, each with named protagonist (Aminata), numbered steps, an explicit **Climax** beat, and verbatim UJ labels in headers.
### Findings
- **medium** Flow 2 (UJ-2, manual entry) has no failure path, even though FR-5's testable consequence ("le montant ne peut pas être nul ou négatif") implies a validation-error state that a downstream engineer must build. EXPERIENCE.md State Patterns also has no row for "invalid amount entry" in the Saisie manuelle bottom sheet. (EXPERIENCE.md Flow 2, lines 156–169; State Patterns table, lines 79–96). *Fix:* add a one-line failure note to Flow 2 ("Cas d'erreur : montant à 0 → CTA Enregistrer désactivé") and a matching State Patterns row.
- **low** Flow 3 (UJ-3, dashboard) has no failure/edge path stated, e.g. what happens if the user opens the Tableau de bord with zero transactions ever recorded (distinct from "mois sans données," which IS covered in State Patterns). PRD UJ-3 also defines no error case, so this is a minor completeness gap rather than a missed requirement. *Fix:* optional — note the all-time-empty case explicitly if it differs from "mois sans données."
- **high** No Key Flow (and no UJ in the PRD) covers Gestion Catégories (FR-13–FR-16: create/rename/delete custom category) even though it's a fully specified feature with four FRs and explicit testable consequences (uniqueness validation, cascade rename, reassignment to "Autre" on delete). The PRD itself doesn't define a UJ for this feature, so the gap originates upstream, but EXPERIENCE.md does not compensate with a Key Flow or even Component/State Patterns rows (see Finding in §3 and §4). *Fix:* add a Key Flow (even a short one) or at minimum full Component/State Patterns rows for category management, since FR-13–16 are in-scope MVP requirements that an engineer will build with zero behavioral guidance otherwise.

## 2. Token completeness — strong
Checked: every token defined in DESIGN.md YAML frontmatter (`colors`, `typography`, `rounded`, `spacing`, `components`) against every `{path.to.token}` reference in DESIGN.md prose and in EXPERIENCE.md. All `{colors.*}`, `{rounded.*}`, `{spacing.*}`, `{components.*}`, `{typography.*}` references resolve to a defined frontmatter key. All 22 color tokens (including the 11 category bg/fg pairs) carry hex values — no missing hex. Contrast target (≥ 4.5:1) is stated once globally in EXPERIENCE.md Accessibility Floor, deferring to DESIGN.md per the inheritance convention, but DESIGN.md itself never restates the numeric contrast target for its specific load-bearing pairs (e.g., `text-secondary` on `surface`, `danger`/`success` on `bg`).
### Findings
- **low** DESIGN.md never states a contrast ratio number anywhere — the 4.5:1 floor lives only in EXPERIENCE.md (Accessibility Floor) and PRD §5 NFRs. For a dark-on-dark palette with a lavender secondary text color (`#A8A8C0` on `#0D0F1E`/`#181B33`), a downstream engineer building dark-mode-only UI would benefit from DESIGN.md flagging which pairs were verified, since DESIGN.md is the file that owns visual contrast per EXPERIENCE.md's own cross-reference ("Le contraste visuel... vit dans DESIGN.md", line 110). *Fix:* add one sentence to DESIGN.md.Colors confirming `text-secondary`/`bg` and `text-secondary`/`surface` meet 4.5:1, or flag if unverified.
- **low** `{nom}` placeholder in `cat-{nom}-bg` / `cat-{nom}-fg` (DESIGN.md lines 87–90, 209) is template syntax, not a resolvable token — harmless but worth a one-line note distinguishing it from the `{path.to.token}` convention, since it's visually identical to a broken reference at first glance.

## 3. Component coverage — thin
Checked: every component name in DESIGN.md.Components (prose) and DESIGN.md frontmatter `components:` YAML against every row in EXPERIENCE.md.Component Patterns, and vice versa.

DESIGN.md visual specs (prose): Carte Solde, FAB, Bottom sheet « Nouvelle transaction », Pastille de catégorie, Ligne de transaction, Ligne de Reçu, Skeleton de chargement, Bannière info, État vide.
DESIGN.md YAML `components:` keys: card-solde, bottom-sheet, category-badge, fab, button-primary, nav-bar, input, segmented-control, warning-row, info-banner.
EXPERIENCE.md Component Patterns rows: Carte Solde, FAB, Bottom sheet Saisie manuelle, Sélecteur Catégorie, Sélecteur Date, Ligne de Reçu, En-tête sticky Reçu, Skeleton chargement, Ligne de Transaction, Entrée Reçu groupé, Sélecteur de Période, Bannière info.
### Findings
- **high** `Sélecteur Catégorie` (EXPERIENCE.md Component Patterns row, line 67) has no DESIGN.md visual spec. DESIGN.md specs only the category *badge/pastille* (the 40px circle), not the bottom-sheet list selector itself (layout, row height, predefined-vs-custom visual grouping). *Fix:* add a DESIGN.md.Components entry for the selector sheet, or fold it explicitly into the "Bottom sheet" generic spec with a note.
- **high** `Sélecteur Date` (EXPERIENCE.md line 68) has no DESIGN.md visual spec at all — no row, no token reference. A native date-picker is invoked but its bottom-sheet wrapper (padding, header, CTA) is unspecified. *Fix:* add a DESIGN.md.Components row, even if it just says "inherits Bottom sheet base spec, native date-picker widget below."
- **medium** `En-tête sticky Reçu` (EXPERIENCE.md line 70) has no DESIGN.md visual spec — typography, background, and sticky-positioning are unstated. DESIGN.md.Typography mentions "total du reçu en en-tête sticky" once in passing (line 162) but never specs the container. *Fix:* add a short Components entry.
- **medium** `Entrée Reçu groupé` (EXPERIENCE.md line 73, expand/collapse grouped receipt rows in Historique) has no DESIGN.md visual spec — chevron icon, expanded-row indentation/background, and animation are all unspecified visually (only "chevron animé" mentioned behaviorally). *Fix:* add a Components entry covering the chevron and expanded-state visual treatment.
- **medium** `Sélecteur de Période` (EXPERIENCE.md line 74, month-nav chevrons + custom range bottom sheet on Tableau de bord) has no DESIGN.md visual spec. *Fix:* add a Components entry for the period-nav row and the custom-range bottom sheet.
- **medium** `nav-bar` exists as a DESIGN.md YAML component (height, colors, border) but has no EXPERIENCE.md Component Patterns row describing its behavior (tab switching, active-state persistence, whether FAB-tap interrupts current tab). It's only ever mentioned as a location reference. *Fix:* add a Component Patterns row for the bottom tab bar.
- **low** `input` exists as a DESIGN.md YAML component (generic text field spec) but has no EXPERIENCE.md behavioral row — its behavior is scattered piecemeal across "Bottom sheet Saisie manuelle" (clavier numérique) and "Ligne de Reçu" (inline editable). This is defensible as "input is a primitive referenced by other components," but isn't stated as a deliberate inheritance decision anywhere.
- **low** `segmented-control` and `warning-row` are DESIGN.md YAML components with real visual specs; their behavior is folded into other rows ("Bottom sheet Saisie manuelle" for segmented-control, "Ligne de Reçu" for warning-row) rather than getting their own row. This is reasonable consolidation, not a true miss — flagged only for completeness.
- **critical** No DESIGN.md visual spec and no EXPERIENCE.md Component/State Patterns exist for the **Gestion Catégories surface** (create/rename/delete custom category UI) despite FR-13–FR-16 being fully scoped MVP requirements with explicit testable consequences (uniqueness validation message, cascade rename, delete-confirmation, reassignment to "Autre"). Neither file names a single component for this surface (no "category list row," no "create category input," no delete-confirmation dialog spec). *Fix:* this is the single largest gap in the pair — add minimum-viable Component Patterns + visual spec rows for: category list row, create/rename input, delete confirmation.

## 4. State coverage — adequate
Checked every IA surface (EXPERIENCE.md Information Architecture table) against EXPERIENCE.md State Patterns for empty/cold-load/focus/error/offline/permission-denied states where applicable.
### Findings
- **critical** **Gestion Catégories** (IA row, line 35) has zero State Patterns coverage: no empty state (0 custom categories), no duplicate-name error state (FR-14 requires case-insensitive uniqueness validation), no delete-confirmation state (FR-16 requires confirmation), no "predefined category — action blocked" state (FR-15/16 explicitly forbid rename/delete on predefined categories — the UI must communicate this, e.g., disabled controls or hidden actions). *Fix:* add 3–4 State Patterns rows for this surface.
- **medium** **Paramètres** (IA row, line 36) has zero State Patterns coverage — not even a basic row listing its content (déconnexion, accès catégories, avertissement stockage). Given it's a static settings list this is lower severity than Gestion Catégories, but still has no documented row at all.
- **low** **Détail Transaction** has a "Suppression Transaction" state (line 96) but no explicit "edit mode" state for FR-7 (modify any field of an existing transaction) — FR-7 is a full MVP requirement (montant, Catégorie, date, note all editable) and the only EXPERIENCE.md trace of it is the IA row description ("Édition ou suppression"). No State Patterns row, no Component Patterns row, no Key Flow walks through editing an existing transaction. *Fix:* add a State Patterns row or fold edit-mode behavior into the "Ligne de Transaction" or a new "Détail Transaction" Component Patterns row.
- **low** **Scan — Capture** has "Réseau absent" (line 89) and implicitly a permission-prompt state (mentioned only in Key Flow 1 step 5: "L'app demande la permission caméra"), but no dedicated State Patterns row for camera-permission-denied (as opposed to not-yet-asked). FR-9 requires the app to "explain why" on first access — there's no row for what happens if the user denies. *Fix:* add a State Patterns row for permission-denied (camera/gallery).
- **strong (no finding)** Auth, Accueil, Scan—Revue, Historique, Tableau de bord all have thorough state coverage including cold-open, empty, loading, timeout, error, and edge-value states (e.g., "mois sans données," "variation — mois précédent vide").
