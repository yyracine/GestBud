# Accessibility Review — GestBud

## Overall verdict

This spine pair is better than average for a v1 fintech spec — it names a contrast floor, a touch-target floor, and Reduce Motion explicitly, and it makes good calls (no swipe-to-delete on Historique, confirmation-gated deletion, plain-language errors). But it ships real gaps that matter because this app touches people's money on cheap Android phones, often for first-time formal-finance users: two category badge pairs fail WCAG AA outright, the "Solde courant" label at 70% opacity is mathematically close to or below the 3:1 floor, color is the *only* signal for dépense/revenu sign almost everywhere (no +/− prefix specified), several named interactive patterns (segmented control, expandable receipt rows, swipe-to-delete, FAB menu) have zero concrete screen-reader announcement text, and border/divider tokens are nearly invisible (1.1–1.4:1) which will hurt low-vision users distinguishing list rows and input fields. None of this is exotic to fix, but as written a screen-reader user cannot reliably tell income from expense, cannot tell an expanded receipt from a collapsed one, and on two categories cannot read the badge color reliably even with normal vision.

## Findings

- **critical** Two category badge fg/bg pairs fail WCAG AA for text/icon-scale content: `cat-education` (`#7E57C2` on `#1E1A3A`) = **3.19:1**, and `cat-habillement` (`#AB47BC` on `#2A1A3A`) = **3.33:1**. Both are below 4.5:1; since the icon is 20px (DESIGN.md component spec) it does not clearly qualify as "large text" under WCAG, and even under the lenient 3:1 UI-component threshold, `cat-education` is marginal and offers no margin for low-end-display gamma drift. (DESIGN.md lines 30-31, 34-35, 87-90). *Fix:* Lighten `cat-education-fg` to something like `#9575CD`→`#A287D6`-class and `cat-habillement-fg` toward `#C158DC`-class until both clear 4.5:1 against their current bg, or darken/desaturate the bg tokens instead — re-verify computationally, don't eyeball.

- **critical** Color is the only signal distinguishing dépense (red) from revenu (green) everywhere in the spec — Carte Solde variation, Ligne de Transaction amount, Tableau de bord postes, the `↑ +100%` Santé indicator in Flow 3. No +/− prefix, no icon, no textual "Dépense"/"Revenu" repeated near the amount is specified as a redundant cue. This fails WCAG 1.4.1 (Use of Color) and directly harms the ~8% of men with red-green color blindness — plausible at scale for "urban professionals." (DESIGN.md lines 151-152, 166, 230; EXPERIENCE.md lines 64, 181). *Fix:* Add a mandatory non-color cue: prefix amounts with `+` / `−`, or repeat the category-badge convention (Revenu has its own badge fg/bg already — lean on it) consistently next to every colored amount, not just on the dedicated Revenu category badge.

- **high** "Solde courant" label is specified at 70% opacity over the accent gradient (`{colors.accent} → #4A3FD4`), which is itself the single most prominent number on the Accueil/Dashboard. Computed: white-at-70%-over-`#6B5CFF` blends to ~3.05:1, white-at-70%-over-`#4A3FD4` to ~4.32:1 — both below 4.5:1 for what is effectively a caption-sized label, and the lower end of the gradient sits right at the 3:1 floor with zero margin for OLED/sunlight viewing conditions common on Android in Dakar/Abidjan. (DESIGN.md line 203: "label Caption « Solde courant » à 70% d'opacité"). *Fix:* Either drop the opacity reduction (use full `text-primary` or a fixed lighter token) or raise opacity to ≥85% and re-verify against both ends of the gradient, since the card is gradient not flat.

- **high** No screen-reader/announced-state guidance exists for several Component Patterns named in EXPERIENCE.md's own table: **segmented control** (Dépense/Revenu — no mention of how TalkBack/VoiceOver announces selected state, e.g. "Dépense, sélectionné, 1 sur 2"), **expandable receipt rows in Historique** (chevron animates visually but no "Reçu, 5 articles, réduit/développé" announcement is specified — the Accessibility Floor only mentions "rôle et état" generically, never ties it to this specific pattern), **swipe-to-delete on Ligne de Reçu** (no specified alternative action for screen-reader users who cannot perform a swipe gesture — VoiceOver/TalkBack typically require a custom action exposed via accessibility actions, not mentioned at all), and **FAB menu** (2-option bottom sheet — no mention that focus must move into the sheet on open, though the general "bottom sheets piègent le focus" rule technically covers it). (EXPERIENCE.md lines 62-75 vs. 108-117). *Fix:* Add one line per pattern to the Accessibility Floor table (or a dedicated sub-table) specifying exact announced strings/states: segmented control selection, expand/collapse state, and — critically — a swipe-to-delete alternative exposed as an accessibility action ("Supprimer cette ligne") since swipe gestures are frequently unreachable for screen-reader and switch-control users.

- **high** Swipe-to-delete on Ligne de Reçu (Scan — Revue) has no stated reachable alternative for users who cannot perform the gesture (motor-impaired, screen-reader users navigating by swipe-to-navigate which conflicts with the same gesture vocabulary). The spec correctly rejected swipe-to-delete on Historique for *financial-risk* reasons but didn't address it for *accessibility* reasons on the surface where it remains. (EXPERIENCE.md line 102, 69). *Fix:* Add a visible/long-press/menu-button alternative to delete a receipt line, and expose it as a VoiceOver/TalkBack custom action regardless.

- **medium** `border` token (`#2A2D4A`) against `bg` (`#0D0F1E`) computes to **1.42:1**, and against `surface-raised` (`#1E2240`) to **1.16:1** — both far under the 3:1 floor WCAG 1.4.11 sets for UI-component boundaries (input borders at rest, list dividers). This token is used for divider lines between every transaction row and the resting border of every input field (DESIGN.md lines 19, 108-114, 174). At rest, low-vision users and anyone on a low-end/glare-affected Android display will not be able to perceive row boundaries or where an input field starts/ends until focus changes the border to accent. *Fix:* This may be an intentional "quiet until interactive" choice, but it should be a documented decision, not silent — confirm whether row separation also relies on a second cue (background-tint banding, spacing) per DESIGN.md's elevation system; if not, lighten `border` or add a secondary boundary cue.

- **medium** Touch target table (EXPERIENCE.md line 113) states "≥ 44 pt (iOS) / 48 dp (Android) pour tous les éléments interactifs y compris... le × de la bannière," but DESIGN.md never gives the × dismiss icon a hit-area spec — only `bottom-sheet` close icon and banner × are mentioned as visual elements with no explicit touch target box (typically these render as a small icon glyph well under 44pt if the tappable area isn't padded out). Same gap for the Ligne de Reçu's swipe-action affordance and the chevron in expandable receipt rows — DESIGN.md gives icon-size (20px) but never a minimum *tap target* independent of icon size. (DESIGN.md lines 205-219; EXPERIENCE.md line 113). *Fix:* Add explicit hit-slop/min-tap-box guidance in DESIGN.md component tokens for every icon-only interactive element (×, chevron, swipe-action button), not just rely on the blanket EXPERIENCE.md rule — implementers will reach for icon-size as tap-size by default.

- **medium** Skeleton/pulse loading animation's Reduce Motion fallback is specified ("passe... à un fond statique") but EXPERIENCE.md doesn't say whether the *screen transitions* (cross-fade) are also suppressed entirely under Reduce Motion, or merely shortened — "transitions cross-fade courtes... plutôt que les slides" reads as the *default* behavior, not explicitly gated behind the Reduce Motion flag. It's ambiguous whether cross-fade is itself disabled (instant cut) when the OS-level Reduce Motion setting is on. (EXPERIENCE.md line 115). *Fix:* Clarify: is cross-fade the unconditional default (replacing slides always), or is it specifically the Reduce-Motion-on behavior with slides being the Reduce-Motion-off default? As written it conflates "our default choice" with "accessibility accommodation."

- **medium** No mention of haptic/sound feedback redundancy for critical confirmations (transaction saved, transaction deleted, receipt validated) for low-vision or attention-impaired users who may not see a transient toast/snackbar-style confirmation. "Reçu enregistré !" and "Transaction enregistrée." are described as brief visual feedback (Flow 1 step 12, Flow 2 step 9) with no mention of how a screen-reader user is notified (live region? focus moves to confirmation? announced via semantics?). (EXPERIENCE.md lines 150, 168). *Fix:* Specify that success microcopy is exposed as an accessibility live-region announcement, not just a visual toast, given these are financial-state-changing confirmations.

- **low** Currency labeling is good in the **microcopy** examples ("Il te reste 87 200 FCFA ce mois" — full word "francs CFA" specified for screen readers per line 112) but inconsistent in the **Key Flows** prose, which uses bare numbers without units in several spots: Flow 3 step 5 "Alimentation 38 000 · Transport 12 500 · Santé 7 200" has no FCFA suffix per line, relying on a presumed shared header; this is a documentation nit (flows are illustrative prose, not spec) but worth flagging since DESIGN.md's Do/Don't table explicitly forbids omitting currency ("Don't: omettre la devise") — the flows should model the rule they require. (EXPERIENCE.md lines 180; DESIGN.md line 226). *Fix:* Add FCFA to each per-category amount in Flow 3 step 5 for consistency, or note that the screen displays a single shared currency header.

- **low** Thin-space (espace fine insécable) thousands separator screen-reader behavior is *flagged as a risk* but not *resolved*: EXPERIENCE.md line 117 says "vérifier que les lecteurs d'écran lisent bien le chiffre complet" — this is phrased as an open verification task, not a confirmed behavior or a stated fallback if VoiceOver/TalkBack mis-parses the non-breaking thin space and reads "deux cent quarante-cinq... huit cents" as two fragments instead of "deux cent quarante-cinq mille huit cents." For a finance app this ambiguity is consequential (a screen-reader user could mishear an amount as 1/1000th of its value). *Fix:* Resolve this before build, not at QA: either confirm both platforms read U+202F correctly with Flutter's semantics tree in French locale, or set an explicit `semanticsLabel`/accessible-value override per amount widget that spells out the full number unambiguously, decoupled from the visual thin-space formatting.

- **low** Single-language (French-only) scope is stated as a deliberate decision ("Langue : interface en français uniquement (v1)" — EXPERIENCE.md line 117) and is reinforced by the brand voice section in DESIGN.md (tutoiement, French-only messaging table). This is acceptably documented as intentional v1 scope, not a silent gap — no action needed, but note it constitutes a literacy/accessibility boundary for non-Francophone residents in target cities (sizeable Anglophone populations in parts of Douala) that should be tracked as a known v2 backlog item rather than re-discovered later.

- **low** Error microcopy in the Voice and Tone table is good about avoiding jargon in *user-facing* strings ("Ce code a expiré" not "Token invalide"; "Impossible de lire le reçu" not "Erreur réseau (code 503)") — this is a genuine strength worth crediting, not a finding. However, the State Patterns table's OCR timeout description still surfaces a numeric threshold internally ("OCR timeout (> 10 s)") that is implementation detail, not user-facing copy, so it's fine — flagging only to confirm no jargon leak was found in any *quoted* user-facing string. No fix needed; this entry exists for audit completeness.

## Unverifiable / out-of-scope items

- Dynamic Type / largest accessibility text size behavior is asserted ("doit rester lisible sans troncation jusqu'à la taille d'accessibilité maximale du système") but cannot be verified from static specs — depends on actual Flutter layout implementation (text scaling, `Flexible`/`AutoSizeText` usage, fixed-height containers like the 60px transaction row and 48px input that could clip at 200%+ system font scale). Flag for implementation-time testing, not a documentable spec gap.
- Actual rendered contrast on real mid/low-end Android panels (color calibration, blue-light filters, outdoor glare common in Dakar/Abidjan) cannot be computed from hex values alone — token-level ratios above are sRGB-ideal; real-world margin is likely worse, reinforcing that marginal pairs (3.0–4.6:1 range) should be treated as failing, not passing.
- Focus order behavior ("l'ordre de focus suit l'ordre de lecture") is asserted but not testable from the spec — depends on widget tree implementation.
