# Microcopy & Voice Review — GestBud

## Overall verdict
The voice rules themselves (tutoiement, chaud/direct vs neutre/clair, no emojis, espace fine + FCFA) are well-defined and the worked examples in the Voice and Tone table follow them correctly, including well-chosen "à éviter" counter-examples. The bigger risk is not in the copy that exists but in the copy that doesn't: several user-facing moments referenced in Component Patterns and State Patterns (swipe-to-delete confirmation, période custom validation, permission caméra refusée, category management, OTP resend success) have no quoted copy at all, leaving tone decisions to whoever implements them. There's also a recurring warmth asymmetry between the two flagship success messages and a few terminology/format inconsistencies that should be tightened before dev handoff.

## Findings

- **high** "Reçu enregistré !" (warm, exclamation) vs "Transaction enregistrée." (flat, period) — both are the two canonical "success" confirmations in the Voice and Tone table (EXPERIENCE.md lines 48–49), yet one reads chaleureux and the other reads neutral/administrative. The stated rule is "chaud et direct sur les succès" for both. *Fix:* either give Transaction the same warmth ("Transaction enregistrée !" or "C'est noté !") or explicitly document that manual-entry confirmations are deliberately quieter than receipt-scan confirmations (a defensible UX choice, but currently looks like an oversight).

- **high** No copy for swipe-to-delete confirmation on Ligne de Reçu (EXPERIENCE.md line 69, line 102) — Component Patterns and Interaction Primitives both specify the gesture ("Swipe-to-delete pour supprimer une ligne") but no confirmation step or copy is given, unlike Transaction deletion which has explicit confirm copy ("Cette transaction sera supprimée définitivement.", line 96). Given the Anti-patterns section explicitly calls accidental-deletion risk on financial data "trop élevé" for Historique, the same risk logic arguably applies to receipt lines (each line carries a real montant). *Fix:* add a State Patterns row for "Suppression ligne Reçu" with explicit copy, e.g. an inline undo affordance or a one-line confirm ("Ligne supprimée." with brief undo) — decide and document whether it's silent-delete or confirmed-delete.

- **high** No copy for "Date fin < date début bloquée" validation (EXPERIENCE.md line 74) — Component Patterns states the rule exists ("blocked") but never says what the user sees: is the end-date simply non-selectable in the picker (no message needed), or does an error appear? *Fix:* clarify mechanism; if an error message is shown, add a State Patterns row with quoted copy in the established neutre/clair register, e.g. "La date de fin doit être après la date de début."

- **medium** No copy for permission caméra refusée (EXPERIENCE.md Flow 1, line 143; not present in State Patterns table at all) — only the happy path is described ("L'app demande la permission caméra... ou ouvre le viewfinder directement"). iOS/Android permission-denial is a near-certain real-world path (per Accessibility Floor and Foundation, permissions are explicitly called out as one of the only OS-divergent behaviors) yet has zero defined copy or fallback flow. *Fix:* add a State Patterns row, e.g. "Permission caméra refusée | Scan — Capture | Message + CTA « Saisir manuellement » (même pattern que Réseau absent)."

- **medium** Gestion Catégories (créer/renommer/supprimer) has an IA entry (EXPERIENCE.md line 35) but zero Voice/State/Component Pattern coverage — no success copy for "catégorie créée," no confirmation copy for "supprimer une catégorie" (which has higher stakes than deleting a line, since it likely affects existing transactions), no error copy for duplicate-name validation. *Fix:* add Component Patterns + State Patterns rows mirroring the Transaction-deletion pattern (explicit confirm, no silent delete) since this is also "financial data" by the doc's own risk standard.

- **medium** OTP resend has no success confirmation copy — State Patterns "OTP en attente" (line 82) only describes the disabled-button/counter state; nothing documents what the user sees after the 60s elapses and they tap "Renvoyer" successfully (vs. the already-covered invalid/expired and no-network cases). *Fix:* add a brief neutral confirmation, e.g. "Nouveau code envoyé." or rely on the counter resetting silently — but state which.

- **medium** OCR timeout error message is unquoted and ambiguously cross-referenced — State Patterns row "OCR timeout (> 10 s)" (line 87) says only "Message d'erreur" without quoting it, while the adjacent "Réseau absent (scan)" row (line 89) is fully quoted. It's unclear whether timeout reuses the Voice table's "Erreur réseau OCR" copy ("Impossible de lire le reçu. Saisis-le manuellement.", line 51) — timeout and network-absent are different failure conditions and may warrant different wording. *Fix:* quote the timeout message explicitly in the State Patterns row, even if it intentionally reuses the existing Erreur réseau OCR string.

- **medium** Emoji rule worded inconsistently across the two documents, and the ⚠ glyph is used as live UI without being reconciled against the rule. DESIGN.md Brand & Style (line 138) says "Pas d'emojis dans les messages critiques" (general, undefined scope of "critique"). DESIGN.md Do/Don't (line 232) narrows this to a specific list: "Emojis dans les messages d'erreur, de solde négatif, ou de suppression" — OCR warning rows are not in that list. EXPERIENCE.md Voice and Tone (line 44) repeats the general "Pas d'emojis dans les messages critiques" phrasing. Meanwhile both DESIGN.md (line 213) and EXPERIENCE.md (line 88) specify an actual ⚠ icon on the OCR warning row. *Fix:* clarify in one place whether the no-emoji rule covers functional/status icons (⚠) or only emoji-as-decoration in copy text — recommend stating explicitly: "La règle s'applique au texte des messages, pas aux icônes système d'état (⚠, ✕, ◀▶)."

- **low** Currency label inconsistency: Accessibility Floor (EXPERIENCE.md line 112) quotes the screen-reader string as "245 800 francs CFA" (spelled out), while every other quoted amount in both documents uses the "FCFA" abbreviation per the DESIGN.md Typography rule (line 166: "La devise `FCFA` est toujours en `caption` inline"). This may be an intentional TTS-friendly expansion, but it's not flagged as an exception anywhere. *Fix:* add a one-line note in Accessibility Floor: "Le suffixe FCFA est développé en « francs CFA » uniquement pour la lecture VoiceOver/TalkBack — l'affichage visuel reste `FCFA`."

- **low** Terminology drift: "Reçu" vs. "ticket" — Information Architecture, Component Patterns, State Patterns, and Voice and Tone all consistently use "Reçu" as the formal product term (Scan Reçu, Ligne de Reçu, Reçu enregistré). Flow 1's narrative (EXPERIENCE.md lines 135, 137, 144, 150) uses "ticket" four times ("scanne son ticket Auchan," "ticket de 38 000 FCFA," "pointe son téléphone sur le ticket," "range son ticket") and never says "reçu" in the prose, only in the quoted UI copy. This is plausibly intentional (diegetic narration describing the physical paper object vs. the in-app data entity) but a downstream reader skimming flows could conflate the two terms or introduce "ticket" into UI copy. *Fix:* add a one-line glossary note: "Ticket = objet physique (langage courant) ; Reçu = terme produit utilisé dans toute l'UI." Or replace "ticket" with "reçu" in flow narration for strict consistency.

- **low** "Opération réussie ✓" appears only as a documented anti-example (À éviter column, EXPERIENCE.md line 48) — confirmed correctly used as a contrast case, not live copy. No action needed, but flagging for completeness since it's the only emoji-bearing string in either document and a fast skim could mistake it for sanctioned copy. *Fix:* none required; optionally bold or italicize "À éviter" column entries containing symbols to make the contrast unmistakable at a glance.

- **low** Ellipsis/punctuation style not fully normalized across loading states — "Analyse du reçu en cours…" (Voice table, line 56, and State Patterns line 86) uses a true ellipsis; no other loading or in-progress copy exists to compare against, so this is a single data point rather than a contradiction, but the doc doesn't state whether "…" vs "..." is the house style. *Fix:* add a one-line style note (e.g., "Ellipsis unicode `…`, jamais trois points `...`") to prevent inconsistent implementation across future loading states (e.g., a future "Chargement…" state for category list, settings, etc.).
