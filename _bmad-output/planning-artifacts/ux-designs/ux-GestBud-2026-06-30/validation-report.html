GestBud — UX Design Validation Report

_bmad-output/planning-artifacts/ux-designs/ux-GestBud-2026-06-30/

La paire de spines est solide sur ses fondamentaux : les trois flows clés (UJ-1/2/3) sont pleinement réalisés avec protagoniste nommé, étapes numérotées et beat climax ; tous les tokens résolvent ; la discipline d'héritage (sources, noms de composants, glossaire) est propre. Le gap critique structurel est la **Gestion Catégories** (FR-13–FR-16) : une fonctionnalité MVP entièrement scopée n'a aucune spec visuelle ni comportementale dans aucun des deux fichiers — un développeur n'a rien pour construire le CRUD de catégories.

Les deux lenses additionnelles déplacent sensiblement le tableau. L'accessibilité relève deux problèmes critiques pour une app financière grand public sur Android d'entrée de gamme : deux pastilles de catégorie échouent le contraste WCAG AA, et la couleur est l'unique signal dépense/revenu partout dans l'app — sans repère non-coloré, ce qui exclut les utilisateurs daltoniens de leurs propres finances. Le microcopy review confirme que les règles de voix sont saines mais que plusieurs moments utilisateurs réels (confirmation de suppression de ligne de reçu, validation de période invalide, refus de permission caméra, gestion de catégories) n'ont aucun texte spécifié — un vide que l'implémenteur devra combler sans guidance.

Accessibility review

review-accessibility.md

Plus rigoureux que la moyenne d'un spec fintech v1 (seuils de contraste et de cible tactile nommés, section Reduce Motion, suppression confirmée). Mais expédie de vrais gaps qui comptent pour une app qui touche l'argent réel d'utilisateurs sur Android d'entrée de gamme, souvent novices de la finance formelle.

Critical

Deux paires badge catégorie échouent WCAG AA

DESIGN.md lignes 30-31, 34-35, 87-90
cat-education (#7E57C2 sur #1E1A3A) = 3.19:1 ; cat-habillement (#AB47BC sur #2A1A3A) = 3.33:1. Les deux sous 4.5:1, marginal même sous le seuil UI 3:1.
Fix: Éclaircir les tokens fg ou assombrir/désaturer les bg jusqu'à dépasser 4.5:1 — vérifier par calcul, pas à l'œil.
Critical

La couleur est l'unique signal dépense/revenu partout dans l'app

DESIGN.md lignes 151-152, 166, 230 ; EXPERIENCE.md lignes 64, 181
Aucun préfixe +/−, aucune icône, aucun rappel textuel redondant. Échoue WCAG 1.4.1 ; affecte directement les ~8% d'hommes daltoniens rouge-vert.
Fix: Ajouter un repère non-coloré obligatoire (préfixe +/− sur tous les montants colorés).
High

Label « Solde courant » à 70% d'opacité sous le seuil de contraste

DESIGN.md ligne 203
Calculé : blanc 70% sur #6B5CFF ≈ 3.05:1, sur #4A3FD4 ≈ 4.32:1 — les deux sous 4.5:1, sur le montant le plus important de l'app.
Fix: Supprimer la réduction d'opacité ou la relever à ≥85% et revérifier sur les deux extrémités du gradient.
High

Aucune guidance lecteur d'écran pour segmented control, lignes reçu expandables, swipe-to-delete, menu FAB

EXPERIENCE.md lignes 62-75 vs 108-117
L'Accessibility Floor ne mentionne « rôle et état » que génériquement, sans le rattacher à ces patterns spécifiques.
Fix: Ajouter une ligne par pattern précisant les chaînes/états annoncés.
High

Swipe-to-delete sans alternative accessible

EXPERIENCE.md lignes 69, 102
Aucune alternative atteignable pour les utilisateurs de lecteur d'écran ou à mobilité réduite qui ne peuvent pas effectuer le geste.
Fix: Ajouter une alternative visible (bouton/menu) et l'exposer comme action d'accessibilité VoiceOver/TalkBack.
Medium

Token border quasi invisible (1.16–1.42:1)

DESIGN.md lignes 19, 108-114, 174
Très sous le seuil 3:1 WCAG 1.4.11 pour les limites de composants UI — utilisé pour tous les diviseurs de liste et bordures de champ au repos.
Fix: Confirmer si un second repère existe (teinte de fond), sinon éclaircir le token.
Medium

Cibles tactiles des icônes seules (×, chevron) non garanties ≥44pt/48dp

DESIGN.md lignes 205-219 ; EXPERIENCE.md ligne 113
DESIGN.md donne une taille d'icône (20px) mais jamais de zone de tap minimale indépendante.
Fix: Ajouter une guidance hit-slop explicite par élément interactif icône-seule.
Medium

Reduce Motion : ambiguïté sur le cross-fade par défaut vs accommodation

EXPERIENCE.md ligne 115
Pas clair si le cross-fade est le défaut inconditionnel ou spécifiquement le comportement Reduce-Motion-on.
Fix: Clarifier la relation entre choix par défaut et accommodation d'accessibilité.
Medium

Pas de redondance haptique/sonore pour les confirmations critiques

EXPERIENCE.md lignes 150, 168
« Reçu enregistré ! » et « Transaction enregistrée. » décrits comme feedback visuel transitoire sans mention d'annonce live-region pour lecteur d'écran.
Fix: Spécifier l'exposition en live-region d'accessibilité, pas seulement visuelle.
Low

Lecture de l'espace fine par les lecteurs d'écran non résolue, seulement flaggée

EXPERIENCE.md ligne 117
Risque de mauvaise lecture d'un montant (ex. fragmenté en deux nombres) — conséquent pour une app financière.
Fix: Résoudre avant le build via semanticsLabel explicite, pas à la QA.

Microcopy & voice review

review-microcopy.md

Les règles de voix sont bien définies et les exemples de la table Voice and Tone les suivent correctement. Le risque principal n'est pas dans le copy existant mais dans le copy manquant : plusieurs moments utilisateurs réels n'ont aucun texte spécifié.

High

Asymétrie de chaleur entre les deux messages de succès canoniques

EXPERIENCE.md lignes 48-49
« Reçu enregistré ! » (chaleureux) vs « Transaction enregistrée. » (neutre/administratif) — la règle déclarée est « chaud et direct » pour les deux.
Fix: Harmoniser la chaleur, ou documenter explicitement la différence comme choix délibéré.
High

Aucun copy pour la confirmation de suppression d'une ligne de reçu

EXPERIENCE.md lignes 69, 102
Le geste swipe-to-delete est spécifié mais sans étape de confirmation, contrairement à la suppression de transaction qui a un copy explicite.
Fix: Ajouter une ligne State Patterns avec copy explicite (confirm ou undo).
High

Aucun copy pour la validation « date fin < date début bloquée »

EXPERIENCE.md ligne 74
Le mécanisme (date non-sélectionnable vs message d'erreur) n'est pas précisé.
Fix: Clarifier le mécanisme ; si message, l'ajouter en State Patterns dans le registre établi.
Medium

Aucun copy pour le refus de permission caméra

EXPERIENCE.md Flow 1 ligne 143
Seul le chemin nominal est décrit ; le refus est un chemin quasi certain en usage réel.
Fix: Ajouter une ligne State Patterns, ex. même pattern que « Réseau absent ».
Medium

Gestion Catégories sans aucun copy (création, suppression, erreur de doublon)

EXPERIENCE.md ligne 35
Enjeu plus élevé que la suppression d'une ligne puisque ça affecte des transactions existantes.
Fix: Ajouter Component/State Patterns en miroir du pattern de suppression de transaction.
Medium

Aucune confirmation de renvoi OTP réussi

EXPERIENCE.md ligne 82
Seul l'état désactivé/compteur est décrit, pas ce qui se passe après un renvoi réussi.
Fix: Ajouter une confirmation neutre ou préciser une réinitialisation silencieuse.
Medium

Message d'erreur OCR timeout non cité, lien avec « Erreur réseau OCR » ambigu

EXPERIENCE.md ligne 87 vs ligne 51
Contrairement à la ligne « Réseau absent (scan) » adjacente, entièrement citée.
Fix: Citer explicitement le message, même s'il réutilise la chaîne existante.
Medium

Règle anti-emoji formulée différemment entre les deux fichiers, ⚠ non réconcilié

DESIGN.md lignes 138, 232 ; EXPERIENCE.md ligne 44
DESIGN.md a une formulation générale puis une liste restreinte ; l'icône ⚠ utilisée en UI live n'est jamais réconciliée avec la règle.
Fix: Clarifier explicitement que la règle vise le texte, pas les icônes système d'état.
Low

Incohérence de libellé de devise : « francs CFA » (lecteur d'écran) vs « FCFA » (partout ailleurs)

EXPERIENCE.md ligne 112 ; DESIGN.md ligne 166
Probablement une expansion volontaire pour la synthèse vocale, mais non signalée comme exception.
Fix: Ajouter une note d'une ligne précisant l'exception TTS.
Low

Dérive terminologique « ticket » vs « Reçu » dans la narration du Flow 1

EXPERIENCE.md lignes 135, 137, 144, 150
Probablement intentionnel (objet physique vs terme produit) mais risque de confusion en lecture rapide.
Fix: Ajouter une note glossaire d'une ligne distinguant les deux usages.
