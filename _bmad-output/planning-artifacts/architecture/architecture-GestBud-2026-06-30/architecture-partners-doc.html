01 — Vision produit

Remplacer le cahier, pas la banque

L'actif urbain d'Abidjan, Dakar ou Douala gère aujourd'hui ses finances dans un carnet. GestBud remplace ce carnet — sans connexion bancaire, sans abonnement, sans cloud.

Les applications de gestion financière existantes ont été conçues pour des marchés où les comptes bancaires sont connectables par API. En Afrique subsaharienne, ce modèle ne fonctionne pas. GestBud part du ticket de caisse : l'utilisateur photographie son reçu, l'OCR lit chaque article, l'IA propose une catégorie, l'utilisateur valide en 30 secondes.

🧾Scan de reçu

Photo → OCR → catégorisation IA automatique. Un ticket de supermarché mixte décomposé en moins de 30 secondes.

✏️Saisie manuelle

Pour les achats sans reçu. Transport, marché, transfert d'argent — 10 secondes, fonctionne hors connexion.

📊Tableau de bord

Solde en temps réel, postes de dépense par catégorie, comparaison mois/mois — le bilan que le cahier ne peut pas donner.

Cible MVP : 20 beta-testeurs urbains, 8 mois. Validation de deux hypothèses : la précision de l'OCR sur les reçus locaux, et le transfert de l'habitude cahier vers l'app.

02 — Architecture générale

Une app mobile, un proxy léger, aucune base de données cloud

L'architecture de GestBud repose sur un principe simple : les données financières de l'utilisateur ne quittent jamais son appareil. La seule infrastructure côté serveur est un proxy stateless qui sert d'intermédiaire avec les APIs tierces.

03 — Décisions architecturales clés

Pourquoi ces choix, et ce qu'ils protègent

  • 1
    Flutter — une seule base de code, deux stores

    Un seul code source compilé nativement pour iOS (App Store) et Android (Google Play). Pas de WebView, pas de PWA — les performances natives sont requises pour le flux scan temps réel et l'expérience caméra.

  • 2
    Données 100% locales (SQLite via Drift)

    Les données financières sont stockées dans une base SQLite chiffrée dans la sandbox de l'app — inaccessibles aux autres apps, garantie OS. Ce choix élimine les coûts d'infrastructure backend, les problèmes de synchronisation, et simplifie la conformité données privées. Contrepartie assumée : pas de sync multi-appareils en V1.

  • 3
    Proxy BFF (Cloudflare Worker) — clés API hors du binaire

    Aucune clé API tierce (Mindee, Mistral, Africa's Talking) n'est embarquée dans l'APK ou l'IPA. Toute tentative d'ingénierie inverse du binaire ne révèle aucun secret. Le proxy est stateless — il ne stocke aucune donnée utilisateur. Coût : ~0 $/mois sur le tier gratuit Cloudflare pour 20 utilisateurs beta.

  • 4
    Catégorisation IA par Mistral (français natif)

    Mistral est un LLM français, entraîné sur du contenu francophone. Pour les libellés de caisses africaines en français — « PD BOULANG. », « SAVON OMO KG » — la qualité de reconnaissance est supérieure aux alternatives anglophones. Les 20 lignes d'un reçu sont envoyées en un seul appel batch : latence < 2 secondes ajoutées au flux scan.

  • 5
    Authentification par numéro de téléphone (OTP SMS)

    Le marché cible a un taux de possession de smartphone élevé mais un taux de création d'email faible. L'OTP par SMS via Africa's Talking (couverture Sénégal, Côte d'Ivoire, Cameroun) est le mode d'authentification le plus naturel pour cet utilisateur. Aucun mot de passe à gérer, aucun compte email requis.

  • 6
    Architecture multi-devises dès V1 (XOF → XAF en V2)

    Le schéma de données inclut un champ devise configurable dès la première version. L'ajout du Franc CFA BEAC (XAF, Cameroun) en V2 ne nécessitera aucune migration de données cassante — uniquement l'activation d'une locale Dart et une UI de sélection. L'expansion géographique est préparée dans l'architecture, pas reportée.

05 — Sécurité et données personnelles

Un modèle de données construit pour la confiance

GestBud a été conçu dès l'origine avec un modèle de données minimal : seul le numéro de téléphone est collecté. Aucun nom, aucune adresse, aucun email.

Ce que GestBud ne fait pas

L'app ne synchronise aucune donnée financière vers un serveur cloud. Elle ne collecte pas de données comportementales. Elle ne partage aucune information avec des tiers au-delà des photos de reçus envoyées à Mindee pour le traitement OCR.

Point de vigilance pré-lancement beta

La politique de rétention des photos par Mindee doit être vérifiée et documentée avant le lancement beta. Si les images sont conservées après traitement, l'option "no data retention" de Mindee doit être activée ou un engagement contractuel de suppression obtenu.

06 — Flux principal : le scan de reçu

Du ticket de caisse à la transaction en 30 secondes

Le scan de reçu est la fonctionnalité différenciante de GestBud. Voici comment il fonctionne techniquement, et pourquoi chaque étape a été conçue pour rester dans la fenêtre de 30 secondes :

0 s

Capture caméra

L'utilisateur pointe son téléphone sur le ticket. L'app utilise la caméra native (accès OS, pas de bibliothèque tiers). Photo compressée avant envoi.

0 → 8 s

OCR via Mindee (via proxy BFF)

La photo est envoyée au BFF, qui la soumet à Mindee v2 et poll le résultat jusqu'à résolution. Mindee extrait chaque ligne : libellé + montant. Timeout BFF : 10 secondes.

8 → 10 s

Catégorisation IA par Mistral

Le BFF envoie toutes les lignes en un seul prompt Mistral. Mistral retourne une catégorie pour chaque libellé (« Alimentation », « Hygiène »…). Appel batch : pas d'appel ligne par ligne.

10 s

Affichage et correction

L'app affiche toutes les lignes avec les catégories suggérées. L'utilisateur corrige si besoin (en moyenne 1–2 lignes sur un reçu de 20). Total de la session : 10–25 secondes.

Final

Validation et mise à jour du Solde

Toutes les lignes sont enregistrées en base locale en une transaction atomique. Le Solde se recalcule instantanément. La photo est effacée de la mémoire.

07 — Scalabilité et roadmap technique

Une architecture conçue pour croître

Le MVP cible 20 utilisateurs. Les choix architecturaux anticipent la croissance sans sur-ingénierie prématurée :

🌍Expansion géographique

Le schéma de données supporte plusieurs devises depuis V1. L'ajout du Franc CFA BEAC (XAF) pour le Cameroun en V2 est une activation de configuration, pas une refonte.

🔄Swap fournisseur IA

La catégorisation est isolée derrière une interface dans le BFF. Changer Mistral pour un autre fournisseur ne touche pas l'app mobile — une modification de configuration BFF.

☁️Synchronisation future

L'architecture locale est compatible avec l'ajout d'une couche de sync cloud en V2 (iCloud/Google Drive ou backend propriétaire) sans restructuration du schéma Drift.

🏦Intégration Mobile Money

Wave, Orange Money, MTN MoMo nécessitent des partenariats commerciaux (3–12 mois par pays). L'architecture de transaction est compatible avec des sources de données externes.

Ce qui est intentionnellement hors périmètre V1

Sync cloud des données, connexion API bancaire, export CSV/PDF, notifications push, budgets prévisionnels. Ces fonctionnalités sont documentées et priorisées pour V2 — elles n'ont pas été omises par oubli, mais différées pour valider les hypothèses core du MVP en premier.

08 — Métriques techniques et indicateurs de succès

Ce que nous mesurons pendant la beta

Temps bout en bout scan

Objectif : ≤ 30 secondes du déclenchement caméra à la validation du reçu (réseau 4G normal).

Taux de correction OCR

Pourcentage de catégories suggérées corrigées par l'utilisateur. Indicateur de qualité Mistral sur les reçus locaux.

Adoption du scan

Objectif : > 50 % des transactions enregistrées via scan (vs saisie manuelle). Valide l'adoption du flux principal.

Rétention mensuelle

Objectif : ≥ 90 % de retour le mois suivant sur 8 mois beta. Valide le transfert de l'habitude cahier.
